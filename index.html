Reddit-Scale Code Migration with Multi-Agent AI

A multi-agent system that re-platforms Reddit's GraphQL edge — 40+ high-throughput services, Python → Go, autonomously. Analyze · plan · generate · validate · publish — self-correcting at every step.

Reddit · Backend / AI-Native Engineer

An 18-month multi-team rewrite, compressed into a 3-month autonomous program.

The problem. Reddit's GraphQL serves the read path for every feed, post, comment, and notification — fanning RPCs to dozens of downstream services at Reddit-scale traffic. Python resolvers were the throughput ceiling. The mandate: rewrite every resolver to Go across 40+ services. The plan: 18 months of manual, multi-team work.

What I built. A multi-agent LLM system that owns the migration end to end — migrate <target> ships a reviewed PR. Five phase-isolated agents (Analyze · Plan · Generate · Validate · Publish), hybrid RAG over the monorepo, validation-driven self-correction, and a LoRA-fine-tuned coder for repo-consistent style. Designed and built end to end: controller, agents, MCP server, retrieval, evals, fine-tune, deployment, observability.

Why it works. Phase isolation — a fresh validator is a real adversarial check on the generator. Three nested loops — SDK tool loop, cross-phase retry, K8s backoff — each catches a different failure class. Repo-aware retrieval — RAG over our own code closes the gap from "compiles" to "looks like we wrote it."

1.Architecture Overview

Each migration is a single Kubernetes Job Pod. Kubernetes handles execution, scheduling, isolation, and retries. A thin Postgres ledger retains historical data for dashboards and analytics. Claude, via the Agent SDK, is the reasoning engine at every phase.

Core design principles

  • One K8s Job per migration — Pod-level isolation across filesystem, process, network, and credentials.
  • Pipeline of focused agent phases running sequentially in a single Pod — enables checkpointing, observability, and phase-specific tool access.
  • Shared context via files on the Pod's ephemeral volume — agents read previous phases' outputs, not serialized blobs between API calls.
  • K8s API = source of truth for active jobs; Postgres = historical ledger written on completion.
  • Least-privilege identity per Pod via K8s ServiceAccounts and RBAC.
  • Shared MCP server provides RAG context for code generation and structured validation — agents call it like any other tool.

2.End-to-End Workflow

From a developer's migrate command to an open PR, in six steps.

Figure 3 — Developer → PR flowchart, showing the six-step lifecycle.
  1. Trigger — Developer runs migrate --target payments/resolve_invoice.py or submits via a dashboard. The request hits the Migration Controller's REST API.
  2. Create K8s Job — Controller validates the target, creates a Job manifest with config and credential references, and submits it to the cluster. An initial PENDING record is written to Postgres.
  3. Provision (initContainer) — K8s schedules the Pod. An initContainer clones the monorepo into an emptyDir volume at /workspace and checks out a new branch: migrate/payments-resolve-invoice-<short-id>.
  4. Execute pipeline — The main container runs the five phases sequentially. Each writes output to /workspace/.migration/, and updates the Job's annotations for live observability.
  5. Produce PR — The Publish phase commits, pushes the branch, and opens a PR. The PR URL is written to Job annotations.
  6. Notify — Controller watches for Job completion, writes terminal state to Postgres (status, PR URL, phase logs), and notifies via Slack or GitHub. On failure, the developer gets the phase that failed and why.

Retry semantics. Phases are idempotent. If validation fails, the agent-runner retries Generate → Validate up to 2 times internally (with the validation error fed back as context). If the entire Pod crashes, the K8s Job's backoffLimit controls Pod-level retries. Exhausted jobs are labeled needs-review for manual triage.

3.Agent Interaction Model

Five agent phases. Each is a separate Agent SDK invocation with its own system prompt, tool set, and success criteria. They share state through files in the Pod's /workspace/.migration/ directory.

Tools
git, GitHub CLI.
Output

Success
PR is open, description populated, CI triggered.

Why these boundaries?

  • Analyze and Plan are separated from Generate — the agent reasons about what to do before writing code. Planning catches issues that would otherwise become validation failures.
  • Validate is separate from Generate — it acts as an adversarial check. A fresh validator agent doesn't have sunk-cost bias toward the generated code.
  • Publish is separate — it's the only phase with side effects outside the worktree (push, PR creation). Isolating it makes dry-run mode trivial.

4.Agentic Loops & Self-Correction

Three distinct loops, nested inside each other. Understanding where each lives — and what controls it — is key to understanding how the system self-corrects.

Figure 4 — Three nested loops. The Agent SDK owns Level 1; the orchestrator owns Levels 2 and 3.

LEVEL 1Agent SDK Tool Loop (every phase)

The core agentic loop, built into the Agent SDK. You don't write this — you configure it with a system prompt, tools, and a max-turns limit. The SDK runs it automatically.

Example: a single Generate phase invocation.

think → act → observe 
1. SDK sends system prompt + user message to Claude API
   System: "You are a Go code generator. Read .migration/plan.md for your plan..."
   User:   "Migrate payments/resolve_invoice.py to Go"

2. Claude responds with a tool call:
   → file_read(".migration/plan.md")

3. SDK executes the tool, sends result back to Claude

4. Claude responds with another tool call:
   → file_read("payments/resolve_invoice.py")

5. SDK executes, sends result back

6. Claude responds with a tool call:
   → file_write("payments/resolve_invoice.go", <generated code>)

7. SDK executes, sends result back

8. Claude responds with a text message (no tool call):
   "Generated resolve_invoice.go with 3 resolvers and 2 types."

9. SDK sees no tool call → loop exits → phase complete

Claude decides how many iterations it needs. A simple type migration might take 5–6 tool calls; a complex resolver with multiple files, 20–30.

ConfigPurposeExample
System promptPhase-specific instructions, constraints, output format"Read the plan, generate Go code following existing patterns in /pkg/..."
ToolsWhat the agent can do (least privilege)Generate gets file_read, file_write, file_edit
Max turnsSafety limit on loop iterations50 turns — prevents runaway agents

LEVEL 2Phase Sequence (orchestrator)

The agent-runner entrypoint (your code, not the SDK) calls each phase in order. Straightforward sequential logic — each run_agent() call spins up a fresh Level 1 loop.

agent-runner entrypoint · pseudocode 
def run_migration(target: str):
    update_phase_annotation("analyzing")
    run_agent(phase="analyze", tools=[file_read, grep, glob])

    update_phase_annotation("planning")
    run_agent(phase="plan", tools=[file_read])

    update_phase_annotation("generating")
    run_agent(phase="generate", tools=[file_read, file_write, file_edit])

    update_phase_annotation("validating")
    success = run_agent(phase="validate", tools=[file_read, file_edit, bash])

    if success:
        update_phase_annotation("publishing")
        run_agent(phase="publish", tools=[git, github_cli])
    else:
        raise ValidationFailed(read_file(".migration/validation.md"))

The orchestrator passes context between phases via files, not by carrying over conversation state.

LEVEL 3Cross-Phase Retry Loop

The most important loop for migration quality. When Validate fails, two layers of self-correction kick in before escalating.

Layer A — Intra-phase self-repair (inside Validate's Level 1 loop)

The Validate agent runs go build, sees an error, and fixes it within its own tool loop. Most issues (wrong import, missing error check, typo) resolve here without ever leaving the phase.

Validate agent · self-repair in action 
Validate agent:
  → bash("go build ./...")           → error: undefined: InvoiceResolver
  → file_read("resolve_invoice.go")  → sees missing struct
  → file_edit("resolve_invoice.go")  → adds the struct
  → bash("go build ./...")           → success
  → bash("go test ./...")            → runs tests

Layer B — Cross-phase retry (orchestrator re-invokes Generate)

If the Validate agent exits its Level 1 loop without a passing build, the orchestrator escalates:

retry logic in agent-runner · pseudocode 
for attempt in range(MAX_RETRIES):  # MAX_RETRIES = 2
    run_agent(phase="generate", tools=[...],
              extra_context=read_file(".migration/validation.md"))
    #                        ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    #  Key: validation errors from the previous attempt are fed
    #  into Generate's context so it can reason about what went wrong

    success = run_agent(phase="validate", tools=[...])
    if success:
        break

if not success:
    label_job("needs-review")
    exit(1)

The critical detail. The Generate agent on retry is a fresh Agent SDK invocation with validation errors appended to its input. It doesn't carry over the conversation from the first attempt — it gets a clean context with "here's what you generated, here's why it failed, try again." This avoids sunk-cost thinking.

What each loop handles

Failure typeCaught byExample
Wrong import pathLevel 1 (Validate self-repair)go build error → agent fixes the import
Missing error handlingLevel 1 (Validate self-repair)golint warning → agent adds if err != nil
Wrong type mappingLevel 3 (cross-phase retry)Multiple build errors → re-generate with error context
Fundamental misread of Python codeLevel 3 exhausted → escalateStill fails after retry → needs-review
Pod crash / OOMK8s backoffLimitPod restarts from scratch on new node

Why not one big loop?

  1. Context window pressure — A single loop accumulates every tool call and result. Separate invocations per phase keep each loop lean.
  2. No adversarial check — An agent that generated bad code is biased toward defending it. A fresh Validate agent acts as a genuine second opinion.
  3. No clean retry boundary — You can't tell an agent "forget the last 30 tool calls." Separate invocations give you a natural reset point.
  4. Observability — With separate phases, you see exactly where a migration is stuck. With one loop, you just see "running."

A shared MCP server Deployment serves two concerns: RAG context for code generation (Analyze, Plan, Generate) and structured validation analysis (Validate). Agents call it like any other tool via the Agent SDK's MCP client over HTTP/SSE.

Execution / analysis split

The MCP server does the analysis; the agent does the execution. This keeps the MCP server stateless while the Pod handles anything needing the workspace.

     → bash("go build ./...")              → gets raw stderr

  6. Compare behavior

  7. Fix issues, re-run until green

The agent-runner configures the MCP server as a remote tool source. The SDK handles the MCP protocol — Claude sees MCP tools alongside local tools with no distinction.

agent-runner · MCP wiring 
MCP_SERVER = MCPServer(

    transport="sse",

def run_agent(phase, tools, mcp_tools=[]):

6.RAG Pipeline

The RAG tools exposed by the MCP server (search_schemas, find_go_patterns, get_resolver, find_similar_migrations) are only as good as the index behind them. This section is the production story for that index: how source code becomes retrievable context, what tradeoffs we made, and how it stays fresh as the monorepo moves underneath us.

Why RAG at all? The monorepo is >3M LOC. We can't fit it in a context window, and we don't want generic LLM Go idioms — we want the repo's own patterns: how this team writes resolvers, handles errors, registers schemas, wires tracing. RAG is the bridge between "Claude knows Go" and "Claude knows our Go."

Pipeline at a glance

Figure 5 — RAG pipeline: Parse → Chunk → Embed → Store (Vector DB) → Refresh. Runs as a K8s CronJob; the MCP server reads the index at query time.
Phase 1

Parse

What
Walk the monorepo with language-aware parsers — tree-sitter for Go/Python, graphql-core for SDL.
Extracts
Go functions, structs, interfaces, methods; Python classes & resolver functions; GraphQL types, fields, directives.
Why AST, not regex
We need clean semantic boundaries. Regex chunking splits a function body across two chunks and the LLM gets garbage.
Output
A stream of Symbol records: {kind, name, file, span, signature, docstring, body, imports, refs}.
Phase 2

Chunk

Strategy
Symbol-level chunks, not token-window chunks. One chunk = one resolver / one type / one struct + its methods.
Size guard
If a symbol exceeds ~1,500 tokens, split on inner blocks but keep the signature and docstring duplicated as a header in each piece.
Metadata per chunk
language, service, package, kind, graphql_type, last_modified, commit_sha, linked_schema_path.
Why it matters
Filter pushdown at retrieval time ("only Go resolvers in pkg/payments") is cheap and dramatic for precision.
Phase 3

Embed

Model
voyage-code-3 (1024-dim) — code-specialized, beats general-purpose embedders on code retrieval. Fallback: text-embedding-3-large.
Hybrid signal
Compute BM25 over the same chunks alongside vectors. We retrieve via dense+sparse fusion (RRF) — vectors miss exact identifiers, BM25 catches them.
What gets embedded
Signature + docstring + body, with the file path and symbol kind prepended as a tiny pseudo-header. The path tokens carry real signal.
Throughput
Batched at 128 chunks/request, async; full-repo cold build ≈ 12 min, incremental refresh < 90 s.
Phase 4

Store

Vector DB
Pinecone (serverless) in production — managed, namespace-per-language, metadata filters, sub-100ms p99.
Index layout
One index, three namespaces: go, python, graphql. Each chunk keyed by {repo}:{path}:{symbol}:{sha}.
Sparse companion
BM25 lives in OpenSearch (already used by infra). MCP fans out to both and fuses results via RRF.
Why not pgvector
We use Postgres only as a ledger. pgvector works fine up to ~1M chunks but ops burden (HNSW tuning, IVFFlat rebuilds, replica lag) wasn't worth it given Pinecone's price at our scale.
Phase 5

Refresh

Triggers
(a) GitHub push webhook to main → incremental job per changed file; (b) nightly full reconcile; (c) on-demand migrate --reindex.
Diff strategy
Compare current commit's symbol map vs. last indexed commit_sha. Re-embed only added/changed symbols; tombstone deleted ones.
Atomicity
Upserts use {path:symbol} as the primary key, so re-runs are idempotent. Stale tombstones swept nightly.
SLA
P95 freshness lag < 5 min after merge to main. We page if it crosses 30 min.

Vector DB choice — what we evaluated

OptionStrengthsWhy we did / didn't pick it
Pinecone (serverless)Fully managed, metadata filters, hybrid (dense + sparse) native, sub-100ms p99, scales without opsProduction choice. Zero infra burden, predictable cost at our chunk count (~400k), namespaces map cleanly to language splits.
WeaviateOSS + managed, modules for hybrid + reranking, multi-tenancy, GraphQL APIStrong runner-up. Picked Pinecone for ops simplicity; would revisit if we needed self-hosted for compliance.

The decision in one sentence. Pinecone for dense + OpenSearch for sparse, fused with Reciprocal Rank Fusion at the MCP server. We get hybrid retrieval without standing up new infra and keep a clean exit path to Qdrant if cost or data-residency forces our hand.

Retrieval at query time

MCP server · find_go_patterns implementation sketch 
def find_go_patterns(query: str, filters: dict, k: int = 8) -> list[Chunk]:
    # 1. Embed the query with the SAME model used at index time
    qvec = voyage.embed(query, model="voyage-code-3", input_type="query")

    # 2. Dense retrieval — Pinecone, namespace-scoped, metadata filtered
    dense = pinecone.query(
        namespace="go",
        vector=qvec,
        top_k=k * 4,
        filter={"kind": {"$in": ["resolver", "method", "func"]},
                "service": filters.get("service")},
    )

    # 3. Sparse retrieval — BM25 over the same chunks
    sparse = opensearch.search(
        index="repo-chunks-go",
        query={"match": {"body": query}},
        size=k * 4,
    )

    # 4. Reciprocal Rank Fusion — combine the two ranked lists
    fused = rrf_merge(dense, sparse, k_const=60)

    # 5. Cross-encoder rerank for the top of the fused list
    reranked = cohere.rerank(query, fused[:32], model="rerank-3")

    return reranked[:k]

Why this shape. Dense alone misses exact-identifier queries (OrderResolver). Sparse alone misses paraphrases ("resolver that paginates with cursor"). RRF fuses their ranks without needing calibrated scores. The reranker is a small cross-encoder that pays off on the top-32 — we don't run it over thousands.

Evaluation — how we know it works

Retrieval quality is measured, not vibed. We maintain a golden set of ~250 (query, ideal_chunks) pairs — built from real past migrations and curated by reviewers.

MetricWhat it tells usTarget
Recall@10Did we retrieve the right chunk at all?> 0.92
nDCG@10Are good chunks ranked above noise?> 0.78
MRRHow fast does the first useful chunk appear?> 0.65
End-to-end migration pass-rateDoes better retrieval = better generated code? (north star)monitored per release

The eval suite runs on every embedder/index/prompt change in CI. We won't ship a retrieval change that drops Recall@10 by >1pt without an explicit override.

Interview-ready story arc. Generic LLMs write generic Go. Our generated code had to land cleanly in a 3M-LOC monorepo with strong internal conventions, so we built a code-aware RAG layer: AST-based parsing for clean chunk boundaries, symbol-level chunks with rich metadata, hybrid dense (Pinecone + voyage-code-3) and sparse (OpenSearch BM25) retrieval fused with RRF, a cross-encoder rerank on the top-32, and an incremental push-webhook refresh that keeps freshness lag under 5 minutes. Evaluated against a 250-pair golden set in CI on every change. The outcome: ~80% of migrations land with zero manual edits — the retrieval layer is what closed the gap from "compiles" to "looks like we wrote it."

What to fine-tune on

Training data comes from your monorepo — three sources combined into instruction-formatted examples:

1. Existing Go code (pattern learning)

Input:  "Write a Go resolver for a GraphQL query that returns a paginated
         list of orders with cursor-based pagination"
Output: <actual Go code from your repo that does exactly this>

2. Python→Go pairs (migration learning)

Input:  "Migrate this Python GraphQL resolver to Go:\n\n<Python source>"
Output: <corresponding Go implementation from your repo>

3. Schema→implementation pairs (type mapping)

Input:  "Implement this GraphQL schema in Go using gqlgen:\n\n<schema snippet>"
Output: <Go types, resolvers, and schema registration from your repo>

LoRA training pipeline

Figure 6 — Extract → Format → Train → Evaluate → Publish. Produces a ~50–100MB adapter.

The base model is never modified. The LoRA adapter is a small overlay — you can version adapters and roll back. Retraining is incremental: new code → new pairs → updated adapter.

How it integrates with the Generate phase

The fine-tuned model is exposed through the MCP server as a new tool: generate_go_code. Claude doesn't call the model directly — the MCP server routes to vLLM internally.

Generate phase · using the fine-tuned model 
Claude (Generate agent):

  1. Reads the plan and Python source
     → file_read(".migration/plan.md")
     → file_read("payments/resolve_invoice.py")

  2. Finds existing patterns to follow
     → mcp.find_go_patterns("invoice resolver with DB transaction")

  3. Asks the fine-tuned model to generate code
     → mcp.generate_go_code(
         python_source=<resolver code>,
         schema=<GraphQL schema>,
         pattern_hint="Follow pkg/resolvers/order.go"
       )
     → gets: Go code that follows repo conventions

  4. Reviews and adapts the generated code
     → file_write("payments/resolve_invoice.go", <adapted code>)

Why combine both models?
Claude alone: correct logic, but may not match repo conventions.
Fine-tuned model alone: matches conventions, but limited reasoning.
Combined: repo-consistent code with Claude-quality reasoning.

Why not fine-tune Claude instead?

ApproachProsCons
LoRA on open modelCheap ($10–50/run), small adapter, retrain weekly, full data controlWeaker reasoning than Claude
Claude fine-tuningBest reasoning + your patterns in one modelExpensive, slow iteration, overkill for style
Prompt-onlyZero training cost, always currentContext limits, inconsistent style

8.Deployment Model

All components run in the migration namespace of a single K8s cluster.

ComponentWhatHow
Migration ControllerDeployment (1–2 replicas)Accepts requests, creates Job manifests, watches for completion, writes to Postgres. SA: migration-controller.
Job PodsK8s Jobs (one per migration)Run the 5-phase pipeline. Image bundles Agent SDK + Go toolchain + git. Gets an emptyDir. SA: migration-runner.
MCP ServerDeployment (2 replicas)Shared RAG + validation tools over SSE. Loads vector index from PVC. SA: migration-mcp.
Index BuilderCronJob (nightly or on push)Parses repo, generates embeddings, writes to PVC.
System PromptsConfigMapPhase-specific prompts + Go templates. Mounted read-only. Update without rebuilding.
CredentialsSecretANTHROPIC_API_KEY, GITHUB_TOKEN, SSH keys. Injected as env vars. Pods can't read Secret objects directly.
Codegen ModelDeployment (1 replica, GPU)vLLM serving Qwen2.5-Coder-14B with QLoRA adapter. Called by MCP internally. A100 or A10G.
LoRA TrainingK8s Job (manual / scheduled)Produces ~50–100MB adapter to PVC.
PostgresStatefulSet or managed DBHistorical ledger. Not in the critical path.

Scaling. Add nodes to the cluster. K8s schedules Pods across capacity. Each Pod is CPU-light (mostly waiting on Claude) but needs ~2GB RAM and ~1GB disk. Concurrency is controlled via ResourceQuota on the migration namespace.

9.State & Memory Model

State lives in two places with distinct roles:

State typeWhereRole
Active job statusK8s Job objectSource of truth while running. Native Pending/Running/Succeeded/Failed; phase progress via migration.dev/phase annotation.
Historical recordPostgresWritten on completion or failure. Powers dashboards and audit.

Postgres schema (historical ledger)

Job table 
Job {
  id:           uuid
  k8s_job_name: string          -- e.g., "migrate-resolve-invoice-a1b2"
  target:       string          -- e.g., "payments/resolve_invoice.py"
  branch:       string          -- "migrate/payments-resolve-invoice-abc123"

Postgres only receives terminal states — PENDING, ANALYZING, etc. are the K8s API's job.

Pod-local state (ephemeral volume)

/workspace layout 
/workspace/                          (emptyDir volume)
├── .migration/
│   ├── context.md                   -- Analyzer output
│   ├── plan.md                      -- Planner output
│   ├── validation.md                -- Validator output
│   └── job.json                     -- Job metadata (target, config)
├── (monorepo files...)
└── (generated Go files...)

The .migration/ directory is the inter-agent communication channel. Each phase reads from previous phases' files and writes its own. Intentionally simple:

  • No message bus or shared memory between agents
  • Files are human-readable — kubectl cp them out of a running Pod for debugging
  • Generated code is pushed to the branch before the Pod exits, so nothing is lost on emptyDir reclaim

10.Key Design Decisions

DecisionRationale
K8s Jobs for execution, Postgres for historyK8s handles scheduling, queuing, retries, isolation natively. Postgres adds queryability for dashboards. No Redis or custom worker pool.
Pod = isolation boundaryStronger than git worktrees. Own filesystem, process space, network, credentials. Runaway agent in one Pod can't touch another.
Agent SDK, not raw APISDK manages tool loop, retries, conversation state. Less code.
emptyDir, not PVCWorkspaces are ephemeral. Code is pushed before exit. Auto-cleans.
ConfigMap for system promptsTune behavior without rebuilding images. New Jobs pick up changes immediately.
Files for inter-agent stateSimpler than queue or DB. Human-inspectable.
Sequential phases, not parallel agentsMigration is inherently sequential. Parallelism at the job level, not phase level.
Validate self-repair loopMost failures are small (wrong import, typo). Fixing in-phase avoids expensive re-generation.
Shared MCP server, not sidecarStateless, read-only. RAG returns file pointers; agent reads locally. Validation accepts raw output; agent runs builds locally. One Deployment avoids duplicated memory.
LoRA on open model, not Claude FTClaude handles reasoning; open model handles style. LoRA is cheap, fast to iterate.
Per-migration token budgetsHard limits on tool calls and tokens per phase. Fail fast instead of silently overspending.
Approval gates opt-in, not defaultAutonomy is the default; --approve-plan is available for high-risk targets.

Cost estimation before execution

controller · dry-run estimation 
def estimate_cost(target: str) -> CostEstimate:
    similar_jobs = query_postgres(
        "SELECT tokens_*, status FROM jobs WHERE target SIMILAR TO %s "
        "ORDER BY completed_at DESC LIMIT 20", target
    )
    avg_tokens = mean(sum(j.tokens_* for phase) for j in similar_jobs)
    estimated_cost = avg_tokens * PRICE_PER_TOKEN
    return CostEstimate(
        tokens=avg_tokens,
        cost=estimated_cost,
        confidence=len(similar_jobs)/20
    )

This powers a CLI --dry-run: "Estimated cost: $0.45, based on 18 similar migrations."

14.Human-in-the-Loop Workflow

The system is autonomous but not unsupervised. Developers interact at three points: triggering, reviewing, and providing feedback. Each is designed so the system works without input in the happy path, but humans can intervene at any stage.

Trigger-time controls

CLI flags 
# Default: fully autonomous, creates PR
migrate --target payments/resolve_invoice.py

# Dry run: Analyze + Plan only, shows plan, no code generated
migrate --target payments/resolve_invoice.py --dry-run

# Approval gate: pauses after Plan, waits for human approval
migrate --target payments/resolve_invoice.py --approve-plan

# Cost estimate: shows estimated cost, asks for confirmation
migrate --target payments/resolve_invoice.py --estimate

--approve-plan flow

  1. Analyze → Plan → writes plan.md
  2. Agent-runner pauses, updates Job annotation: migration.dev/phase = "awaiting-approval"
  3. Controller detects the annotation, sends plan to developer via Slack/email
  4. Developer: approves → agent-runner resumes Generate → Validate → Publish.
    Rejects with feedback → feedback written to .migration/plan_feedback.md, Plan re-runs.
    Cancels → Controller deletes the Job.

Review-time workflow

The PR description includes: migration summary (from context.md), what changed and why (from plan.md), validation results, token usage, retry/confidence indicators.

The Publish phase posts review-ready comments highlighting areas needing attention:

PR review comment (generated) 
<!-- migration-agent: review-needed -->
### Areas needing review

1. **Behavioral difference detected** (line 42-58 in resolve_invoice.go):
   Python returns `null` for deleted invoices, Go returns an error.
   The agent chose to match Python behavior, but verify this is correct.

2. **No test coverage for edge case** (line 89):
   The `currency` field has no test for the `nil` case.

3. **Internal library usage** (line 12):
   Used `pkg/db.QueryRow` based on pattern in `pkg/resolvers/order.go`.
   Verify this is the correct DB access pattern for this service.

Feedback flywheel

When a developer modifies a generated PR before merging, that diff is signal. The controller captures it on merge:

  1. GitHub webhook notifies controller on PR merge.
  2. Controller diffs what the agent generated vs what was merged.
  3. If non-trivial, stores the correction in Postgres categorized as style fix, logic fix, missing error handling, wrong pattern.
Feedback typeImprovesHow
Style correctionsLoRA training dataAdd corrected code to next training batch
Logic fixesSystem promptsAdd "common mistakes" to Generate prompt
Pattern correctionsRAG indexBoost correct pattern's relevance
Schema mismatchesValidation toolsAdd rule to validate_schemas

This creates a flywheel: migrations produce PRs → developers fix PRs → fixes improve the system → next migrations are better → developers fix less.

Handling needs-review migrations

When a migration exhausts all retries, the developer gets a notification with what was attempted, where it failed, and a link to the pushed branch (code is preserved even on failure). They can fix manually, or re-trigger with a hint:

re-trigger with developer hint 
migrate --target payments/resolve_invoice.py \
        --hint "The InvoiceStatus enum is in shared/enums.py, not in the resolver file"

The hint is injected into the Analyze phase's system prompt for the retry.

Cancellation

graceful cancellation 
migrate --cancel migrate-resolve-invoice-a1b2

# What happens:
# 1. Controller sends SIGTERM to the Job Pod
# 2. Agent-runner catches signal, writes state to .migration/cancelled.md
# 3. If any code was generated, pushes the branch (preserves partial work)
# 4. Controller marks Postgres record as CANCELLED
# 5. Developer gets notification with link to partial branch

This architecture gets you from "developer types a command" to "PR is open" with a K8s cluster, two container images (agent-runner + MCP server), a GPU node for the codegen model, and a Postgres instance. Autonomous in the happy path but transparent and interruptible — developers can preview plans, set budget limits, review with structured context, provide feedback that improves future runs, and cancel at any point. The hard parts live in the system prompts, the RAG index quality, the validation tool logic, and the feedback flywheel.
